All evaluation, raw-history, and event-history tables were truncated before this test. Transactions were sent at fixed offered rates (an open-loop load — sends are scheduled on a clock and never wait for prior responses, so a slow pipeline shows up as a growing backlog rather than being hidden by back-pressure). Each step ran for 25s; the suite escalates 5 → 10 → 15 → 20 → 25 TPS. At 25 TPS the pipeline could not drain its backlog within the 240s timeout, so the run stopped there per the test's stop condition.

Breaking point Between 20 and 25 requests/second sustained, the pipeline crosses from stable (flat ~30–55 ms latency, zero drops) to unstable (latency ratcheting up in bursts within the step, 6 of 625 transactions never completing inside the 240s drain window, 24 stuck in backlog). This is a hard capacity ceiling, not a gradual slowdown — see the per-transaction trace in §3.

4What's actually saturating — Postgres CPU

Docker container CPU% sampled every ~8s throughout the test (host has 12 cores, so 1200% = full machine). Postgres CPU climbs almost 5× between 5 and 20 TPS, then more than doubles again for a 25% increase in load — it's the resource that runs out first.

Rule engine time — slowest rules
Mean per-rule evaluation time (ms), from Tazama's own prcgTm instrumentation. Rule 021 dominates.
Root cause Rule 021 ("significant number of transactions with the same amount in the last 24 hours") is a velocity check that scans event_history.transaction for the paying/receiving account's recent history. The load test reuses a pool of 200 accounts, so that table — and each account's slice of it — grows every step. Rule 021's cost grows with it: by the 25 TPS step it averages 65 ms and peaks past 890 ms per call, and it is the single largest consumer of Postgres CPU. In production this means the safe throughput ceiling will drop over time as transaction history accumulates, unless that query is indexed/bounded or the velocity check is moved to a pre-aggregated store (Tazama already runs Valkey/Redis for this purpose in other rules).
Separate bug found — not load-related Rules 074 and 075 failed on 100% of transactions at every load level, including the lightest (5 TPS): invalid input syntax for type double precision: "". This is a data-mapping or configuration defect independent of throughput — it means those two typology rules contribute nothing to fraud detection right now regardless of load. Worth a ticket on its own.

5Methodology — what was measured and how

This directly addresses two questions raised going in: whether pacs.002 CreDtTm is usable for timing, and what Tazama already gives you.

Correlation key. Each synthetic transaction's TMI trs_txnid_1 field flows unchanged into the pacs.002 message as OrgnlEndToEndId (src/iso/message-generation.ts:520-521), and Tazama's evaluation record stores that value verbatim inside the stored JSON. That's how each sent message was matched back to its resulting database row.

pacs.002 CreDtTm — checked, not usable for latency. CreDtTm is set at message-generation.ts:517 by copying the inbound message's processing timestamp — it is not stamped when Tazama evaluates the transaction, and it only carries second precision. It tells you when NBP says the transaction happened, not how long Tazama took to process it. It was not used for timing in this report.

What was used instead. The evaluation Postgres table had no timestamp column at all — it only had a JSONB payload keyed on (messageId, tenantId). A inserted_at timestamptz default clock_timestamp() column was added to that table (additive, non-breaking) so each row records the moment Tazama's transaction-aggregation-decisioning processor (TADP) wrote its result. End-to-end latency = that timestamp minus the client's send timestamp, both read from the same host clock.

Tazama's own internal timing (already present, nanosecond precision). Every evaluation record already carries report.metaData.prcgTmED (event-director stage), report.tadpResult.prcgTm (decisioning stage), and a per-rule prcgTm for every rule and the typology processor — all in nanoseconds. This is what powered the rule-level breakdown in §4; no extra instrumentation was needed for it.

Test harness. perf_breakpoint_test.py sends real 1365-byte TMI1910 fixed-width messages straight to the adapter's TCP listener (localhost:3004), varying only account numbers, amounts, and the embedded timestamp/transaction-ID per message. Sends are scheduled on a clock (open-loop), not gated on responses, so an overloaded pipeline shows up as a growing backlog instead of being masked by client-side throttling.

6Is there APM already? What to do about ongoing monitoring

Nothing is running today. A grep across nbp-ppa's source, Tazama's core services, and this deployment turned up no Prometheus, StatsD, New Relic, or OpenTelemetry — the only timing data that exists is the prcgTm fields already inside each evaluation JSON record (see §5), which is why this report had to compute latency from raw DB rows rather than reading it off a dashboard.

But the stack already knows how to wire up Elastic APM — it's just off. Full-Stack-Docker-Tazama/docker-compose.utils.apm-elastic.yaml ships a ready-made apm-server service plus APM_ACTIVE / APM_URL environment overrides for TMS, Event Director, Typology Processor, TADP, and rule-901 — the official images already bundle the Elastic APM agent, it just needs APM_ACTIVE=true and the addon compose file included at deploy time.

Recommendation Enable the built-in addon rather than building custom instrumentation: add -f docker-compose.utils.apm-elastic.yaml (which pulls in docker-compose.utils.elastic.yaml for Elasticsearch/Kibana) to the stack's compose command, or toggle it through tazama.sh. That gives per-service transaction traces and a Kibana APM view for free — no code changes to nbp-ppa or Tazama, and no more reading latency out of raw JSON columns.

8Corroborating run

A second, coarser-grained run (5 → 10 → 20 → 30 → 50 TPS, larger jumps) was run earlier in the same session. It broke later — between 30 and 50 TPS instead of 20 and 25 — because it reached the same load levels with less accumulated transaction history behind it (bigger jumps between steps means less total data written before hitting the higher rates). This is consistent with, not contradictory to, the root cause in §4: the breaking point is a function of both request rate and accumulated per-account history, not request rate alone. It should be read as a second data point, not the headline number — the report above is a clean fine-grained run from a freshly truncated database and is the authoritative one.

Environment: single Docker host, Tazama 3.0.0 full-service deployment (33 rules, Postgres 18, NATS, Valkey), nbp-ppa adapter built from source and run directly with node dist/main. Not a production-sized deployment — absolute TPS numbers are specific to this host's resources (12 cores, Postgres given default full-deployment tuning); the shape of the result — a hard cliff driven by Postgres/rule-021 CPU, worsening as history accumulates — is the portable finding.
